# 씬박스 — 짐 부피(파렛트) 추정 기능 구현 지시서 (PRD)

> 클로드코드 작업용. 기존 씬박스(Next.js / Supabase / Vercel)에 부가 기능으로 추가한다.
> 작업 시작 전 `/init`으로 프로젝트 구조를 먼저 파악하고, 기존 인증·Supabase 클라이언트·라우트·컴포넌트 패턴을 재사용할 것.
> 코드 수정은 플랜 모드로 계획을 먼저 제시하고 승인받은 뒤 진행한다.
> 코드 전달 컨벤션: "찾기→변경" 형식, 개별 `git add "경로"`(따옴표 포함), `git add .` 금지.

---

## 0. 한 줄 정의
씬박스 사용자가 보관할 짐 사진을 올리면, AI가 표준 파렛트(1.1×1.1×2.0m = 2.42㎥) 몇 개 분량인지 범위로 추정해 보여주는 부가 기능. 랜딩 페이지에 진입점을 둔다.

---

## 1. 작업 전 기존 코드 파악 (먼저 보고할 것)
- 인증 방식 (Supabase Auth + Kakao OAuth 여부, 비로그인 접근 허용 여부)
- Supabase 클라이언트 초기화 위치·사용 패턴
- 라우트 구조 (app router / pages router, 랜딩 페이지 파일 위치)
- 환경변수 관리 방식 (Gemini API 키를 둘 위치)
- 기존 컴포넌트·스타일 컨벤션 (mint green 테마, 재사용 UI)

---

## 2. 사용자 플로우
1. 랜딩 페이지에 "내 짐 부피 알아보기" 진입 섹션
2. 추정 페이지: A4 촬영 가이드 → 사진 업로드(다중) + 보조입력
3. "추정하기" → 로딩 → 결과("약 N~M파렛트" + 신뢰도 + 안내문)
4. (선택) 결과 하단에서 씬박스 보관 문의로 연결

---

## 3. 화면 사양

### 3-1. 진입점 (랜딩)
기존 랜딩에 섹션 추가, mint 테마 유지.
카피 예: "보관할 짐, 얼마나 들어갈까요? 사진 한 장으로 알아보세요."

### 3-2. 추정 페이지 (신규 라우트)
- 상단: A4 촬영 가이드 (§6, 아코디언 권장)
- 입력: 사진 다중 업로드(미리보기, 모바일 카메라 촬영 지원) / A4 부착 여부 토글 / 크기 힌트(선택) / 물품 설명(선택)
- 버튼: "추정하기" (사진 1장 이상일 때 활성화)

### 3-3. 결과 영역
- 크게: `약 {min}~{max} 파렛트`
- 작게: 신뢰도 뱃지(높음/중/낮음), advice_to_user 안내문
- 참고 표기: "1파렛트 = 1.1×1.1×2.0m ≈ 2.42㎥, 참고용 추정치입니다."
- objects 상세·reasoning은 화면 미노출 (DB에만 저장)

---

## 4. 백엔드 사양

### 4-1. API 라우트 (신규)
- `POST /api/pallet-estimate` (기존 컨벤션에 맞춰 경로 조정)
- 입력: 이미지(base64 또는 multipart) + 보조입력
- 처리: ① Gemini(gemini-2.5-flash 비전)로 전송, 프롬프트 §5 → ② 코드펜스 제거 후 JSON 파싱(try/catch 폴백) → ③ Supabase 저장(§7) → ④ 노출용 필드만 반환
- Gemini API 키는 서버 환경변수로만 (클라이언트 노출 금지)

### 4-2. 에러 처리
- JSON 파싱 실패 시 1회 재시도 → 실패하면 "추정 실패, 다시 시도" 반환
- 이미지 용량/형식 검증 (대용량 리사이즈, jpg/png/webp 허용)

## 5. Gemini 프롬프트 (그대로 사용)

### 시스템 프롬프트
```
당신은 물류 보관 전문가입니다. 사용자가 창고에 보관하려는 물건의 사진을 받아,
표준 파렛트(1.1m × 1.1m × 2.0m = 2.42㎥) 몇 개 분량인지 추정합니다.

# 스케일 기준 (가장 중요)
- 사진 속에는 보통 크기 기준으로 A4 용지(297mm × 210mm)가 함께 찍혀 있습니다.
- A4 용지를 먼저 찾아, 그 실제 크기(297×210mm)를 기준으로 화면 속 길이를 환산하세요.
  예: A4 긴 변이 사진에서 100px이고 물체 높이가 600px이면, 물체 높이 ≈ 297mm × 6 = 약 1.78m.
- A4가 비스듬히 찍혔으면 원근을 감안해 보정하세요.
- A4가 없으면, 해당 물체의 일반적 표준 크기를 가정하되 신뢰도를 '낮음'으로 내립니다.

# 추정 절차 (반드시 이 순서로 사고)
① 사진 속 A4 용지를 찾는다. (있으면 스케일 기준 확보 / 없으면 표준크기 가정)
② 각 물체가 무엇인지 식별한다.
③ A4를 기준으로 각 물체의 실제 치수(가로·세로·높이 m)를 환산한다.
④ 각 물체의 부피(㎥)를 계산하고 파렛트(2.42㎥)로 나눈다.
⑤ 적재 손실(통상 20~30%, 빈틈·비정형 고려)을 반영해 합산하고 범위로 만든다.

# 원칙
- 결과는 항상 범위로 제시한다 (예: 2~4파렛트). 단일값으로 단정하지 않는다.
- 추정 근거와 신뢰도를 반드시 함께 제시한다. 모르면 모른다고 밝힌다.
- 출력은 아래 JSON 형식만 반환한다. 그 외 설명·서론·markdown 표기를 붙이지 않는다.
```

### 사용자 메시지
```
[이미지 1..N]

보조 정보:
- A4 부착 여부: {부착함 / 안 함}
- 크기 힌트(선택): {텍스트}
- 보관 물품 설명(선택): {텍스트}
```

### 출력 형식 (JSON 고정)
```json
{
  "a4_detected": true,
  "objects": [
    { "name": "철제 캐비닛", "scale_basis": "A4 기준 환산",
      "estimated_dimensions_m": { "w": 0.9, "d": 0.45, "h": 1.8 },
      "volume_m3": 0.73, "pallets": 0.3 }
  ],
  "loading_loss_applied_pct": 25,
  "total_pallets_range": { "min": 2, "max": 4 },
  "confidence": "중",
  "reasoning": "...",
  "advice_to_user": "..."
}
```

```
📄 정확한 추정을 위해 A4 용지를 활용해주세요

② 보관할 물건의 잘 보이는 정면에 A4 용지를 붙이거나 기대 세워주세요.
   - 평평하게, 물건 표면에 밀착시켜 주세요.

④ 물건이 여러 개면, 각 물건마다 A4를 붙여 따로 찍어주세요.

※ A4 없이도 추정은 가능하지만, 범위가 넓어지고 정확도가 떨어집니다.
```

## 7. DB 사양 (Supabase / 검증 루프 핵심)

신규 테이블 `pallet_estimates`:

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid (pk) | |
| user_id | uuid (nullable) | 로그인 시 연결, 비로그인 허용 시 null |
| created_at | timestamptz | |
| image_urls | text[] | Supabase Storage 경로 |
| a4_attached | boolean | 사용자 입력 |
| size_hint | text | 사용자 입력(선택) |
| item_desc | text | 사용자 입력(선택) |
| a4_detected | boolean | Gemini 판단 |
| result_json | jsonb | 전체 응답(objects/reasoning 등) |
| pallets_min | numeric | 노출값 |
| pallets_max | numeric | 노출값 |
| confidence | text | 높음/중/낮음 |
| actual_pallets | numeric (nullable) | 추후 실제 보관 시 입력 → 추정 vs 실제 |
| verified_at | timestamptz (nullable) | 실제값 입력 시각 |

- 이미지는 Supabase Storage 버킷(`pallet-estimates`)에 저장, 경로를 image_urls에 기록
- actual_pallets/verified_at은 이번 구현에선 컬럼만 준비 (운영자가 추후 채움)

## 8. 구현 순서 (이대로 단계 진행, 각 단계 후 확인)

6. dev 서버 실행 → 에러 수정 → 실제 사진으로 동작 확인
7. 커밋 (개별 `git add "경로"`)
